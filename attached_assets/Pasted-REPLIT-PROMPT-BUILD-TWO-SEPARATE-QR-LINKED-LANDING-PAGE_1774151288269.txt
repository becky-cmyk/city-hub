REPLIT PROMPT — BUILD TWO SEPARATE QR-LINKED LANDING PAGE SYSTEMS
PROJECT: CLT City Hub / City Hub
PURPOSE: Micro Publications + Kids Menus Digital Twin Pages
PRIORITY: Build as clean internal systems that can scale by district and issue/month without changing the public QR URL

==================================================
GOAL
==================================================

Build TWO SEPARATE SYSTEMS inside the existing City Hub app:

1. MICRO PUBLICATION LANDING PAGE SYSTEM
2. KIDS MENU LANDING PAGE SYSTEM

These are NOT the same system and should not be merged into one content type.
They may share reusable components, but they must be modeled as separate systems in admin, storage, and routing.

The purpose is to support printed pieces that have QR codes pointing to dedicated online “digital twin” landing pages.

Each printed piece should have a stable QR URL that does NOT change over time.
When a new issue/version is published, that same URL should automatically resolve to the newest live version.
Older versions must remain archived and accessible internally (and optionally publicly through archive links).

==================================================
CORE REQUIREMENT — STABLE QR URL
==================================================

The public QR URL for a district/system must remain constant.

Example concept only:
- /pineville-local-core/publication
- /pineville-local-core/kids
- /district-slug/publication
- /district-slug/kids

Behavior:
- QR always points to the same stable public URL
- public URL always resolves to the CURRENT / most recent published version
- previous versions are archived, not overwritten
- admin can view archive history and switch what is “current”
- archive system must preserve prior sponsor layouts/content records for reference

Do NOT create a system where every new month requires a brand-new QR or a changed public URL.

Build a dedicated publication system for the 6 district publications.

1. Pineville Local Core
2. Pineville Service Corridors

SYSTEM 2 — KIDS MENU LANDING PAGE SYSTEM
==================================================

Build a dedicated kids menu system, separate from publications.

Each printed kids menu (by district) has a QR code that opens its online digital twin.

This page should allow the printed menu to extend online with:
- sponsor continuation
- family fun offers
- digital activities

- giveaway tie-ins
- future kids/family content expansion

The kids menu system must also use stable QR URLs per district.

Behavior:
- same QR always stays valid
- newest published kids menu page is the live page

==================================================
URL / ROUTING RULES
==================================================

Build two routing layers:

1. STABLE PUBLIC ROUTE
This route never changes and is what the QR points to.

2. VERSIONED / ARCHIVE ROUTE
Each version should have its own internal or public issue slug/ID for archive reference.

Examples of desired behavior:
- /pineville-local-core/publication -> resolves to latest current published publication issue
- /pineville-local-core/kids -> resolves to latest current published kids issue
- archive page available internally and optionally publicly:
  - /pineville-local-core/publication/archive
  - /pineville-local-core/kids/archive

Optional individual issue route examples:
- /pineville-local-core/publication/issues/2026-04
- /pineville-local-core/kids/issues/2026-04

But again:
THE QR SHOULD ALWAYS USE THE STABLE PUBLIC ROUTE.

UI / PAGE RULES
==================================================

Public pages should be clean, mobile-first, QR-friendly, and fast.

Because these come from print:
- page must load quickly

- district identity must be clear

Publication pages should feel:
- local
- district-specific
- sponsor-supportive
- useful beyond print

Kids pages should feel:
- family-friendly
- simple
- mobile-first
- sponsor-friendly without clutter
- expandable later for more kids/family content

==================================================
ARCHIVE LOGIC
==================================================

Archive logic must be intentional:

When publishing a new issue:
- old issue remains stored
- old issue is marked archived
- stable QR route points to new issue
- admin can still view/edit archive records if needed
- archived issue should not be deleted

Optional:
- public archive page can show previous issues for district
- if omitted for now, still preserve full archive internally
