OBJECTIVE:
Create a single, centralized entitlement “resolver” and “gate” used by the backend and frontend so that:
1) The app can answer: “What tier is this business currently entitled to?”
2) The app can enforce: “This business cannot access premium features unless entitled”
3) All checks are time-aware (end_at) and status-aware
4) The app stops using any boolean flags (e.g., featured=true) as the source of truth for entitlement behavior (we can leave existing booleans in place for now, but gating must NOT depend on them).

   - Add ONE middleware (or equivalent gate helper) that can protect routes

   - Choose ONE route that is clearly premium-gated (examples: “edit listing extended fields”, “upload gallery media”, “set featured placement”, or similar existing route).

   - Expose tier + allowedCapabilities to the UI in a way the frontend can use.

DEFINITION: “ACTIVE ENTITLEMENT”
An entitlement is active if:
- status indicates active access (use the statuses currently in your entitlements table; include at least: 'active' and 'trialing' if present)
AND
- (end_at is NULL OR end_at > now())
AND
- city scope matches the business’s cityId.

IMPLEMENTATION DETAILS (REQUIRED):
1) Entitlement Resolution:
   - Implement a single query that fetches all entitlements for the business scoped to cityId.
   - Filter to active using the rules above.
   - The resolver must be callable from routes without duplicating query logic.

2) Tier Derivation:
   - Derive tier from product_type values in active entitlements.
   - If multiple, highest wins. Use this strict precedence:
     CHAMBER > PREMIUM > BASIC > FREE
   - If your system uses different product_type strings, map them, but DO NOT rename product types in DB.

3) Capability Map (returned to frontend):
   Return an object like:
   {
     tier: "FREE|BASIC|PREMIUM|CHAMBER",
     capabilities: {
       canUseGallery: boolean,
       canUseMediaBlocks: boolean,
       canShowBadges: boolean,
       canPriorityRank: boolean
     }
   }
   The exact capabilities can be adjusted to match existing features, but must include at least 2 capabilities that are currently present in the UI.

4) Gate Helper:
   - Create a reusable gate:
     requireEntitlement(productType)
   - On failure, respond with:
     HTTP 403
     { error: "ENTITLEMENT_REQUIRED", requiredProductType, tier, businessId, cityId }

5) Proof Wiring (Backend):
   - Apply the gate to ONE real route.
   - Ensure the route still works for entitled businesses.
   - Ensure it blocks non-entitled.

6) Proof Wiring (Frontend):

   - Disable/hide one premium UI control based on capabilities.
   - Display a small text line: “Upgrade to unlock this feature.”
