PART 1 — REFERME
Core Concept: The Referral Triangle
Every referral involves exactly three roles:

C (Connector) = the logged-in user. The person making the introduction.
Person A = the person being referred (they need something — a service, a connection, an opportunity).
Person B = the person or business receiving the referral (they can provide what A needs).
C picks A and B from their own contact list. C writes messages. C tracks outcomes. A and B are notified. The relationship between A and B is what C is creating.

Database Tables
referral_triangles
id              integer, primary key, auto-increment
userId          string (foreign key → users) — this is C, the Connector
personAId       integer (foreign key → contacts) — the person being referred
personBId       integer (foreign key → contacts) — the person receiving the referral
referralType    enum: "b2b" | "b2c" — default "b2b"
hatId           integer, nullable (foreign key → hats) — which business context this referral is made from
sharedMessage   text, nullable — introduction message visible to BOTH A and B
privateMessageToA  text, nullable — private note only A sees

Status Lifecycle
submitted → contacted → connected → in_progress → completed
                                               ↘ declined
submitted — referral created, notifications sent, waiting for B to respond
contacted — B has reached out to A (or C confirmed this)
connected — A and B have met/spoken
in_progress — they are actively working together
completed — the referral led to a successful outcome
declined — A or B declined, or the opportunity did not work out
Status is updated manually by C (the user) via a dropdown on each referral card. When status changes, email and SMS notifications go out to both A and B.

Referral Type
B2B — Business-to-business. C is connecting two business contacts. Branding label: "ReferMe"
B2C — Business-to-consumer. C is connecting a person to a business. Branding label varies.
The referralType field drives email template copy but the mechanics are identical.

Notifications
When a referral is created:

B receives an email: "You've been referred to [A's name] by [C's name]"
A receives an email: "You've been introduced to [B's name] by [C's name]"
Both receive SMS if phone numbers are on file
Each email contains a 1x1 tracking pixel URL (unique token per person per referral) — when the pixel loads, an engagement_event row is created with firedAt set to now, which triggers an engagement nudge for C
When status changes:

Both A and B receive an email update with the new status
Both receive SMS if phone numbers are on file
Respect quiet hours: if the user has quiet hours configured, queue the message until the window opens
The system uses Resend for email and Twilio for SMS.

PART 2 — LIVING CLIENTS (Hats)
Core Concept
"Hats" are business contexts. The user is one person but wears different hats in different professional roles. Each hat is a separate CRM lens for managing clients within that role.

The Doctrine:

A person is one — the same contact can appear in multiple hats
A hat is a context lens — never mix details across hats unless explicitly requested
A person may be a client in one hat and a partner in another

PART 3 — NUDGE SYSTEM
Overview
The nudge system surfaces 8 priority items per day ("daily budget" = 8) for the user to act on. Nudges are computed fresh on every request — nothing is stored. They are scored and sorted by priority score, then capped at 8.

There are 5 nudge types:

Type 1: follow_up
Triggers on contacts who have a nudgeWindowDays setting and whose lastContactedAt is past their window.

Scoring logic:

daysSinceContact = today - (lastContactedAt || createdAt)
overdueDays = daysSinceContact - nudgeWindowDays
overdueDays > 7  → score = 100, urgency = "high"
overdueDays > 0  → score = 80,  urgency = "high"
overdueDays >= -2 → score = 60, urgency = "medium"
overdueDays >= -5 → score = 30, urgency = "low"
otherwise        → no nudge
Modifiers:
  +5 if contact.category === "business"
  +10 if contact.isFavorite
  +0-9 rotation seed (deterministic per contact per day, prevents the same contact showing every day)
Suggested actions:

30 days: "It's been a while — reconnect with a quick hello"

14 days: "Set up a one-to-one catch-up"

Has company: "Touch base about [company name]"
Default: "Check in — see how they're doing"
Snooze system: Contacts have nudgeSnoozeUntil (date) and nudgeSkippedToday (date). If snoozed, skip. If skipped today, skip.

Type 2: birthday
Triggers when a contact's birthday is within a 7-day window (including yesterday = -1 for missed birthdays).

birthdayDaysAway = days until this year's birthday (or last year's if just passed)
today (0 or -1)  → score = 150, urgency = "high",   action = "Send a birthday message or give them a call"
tomorrow (1)     → score = 120, urgency = "high",   action = "Plan a birthday greeting for tomorrow"
2-7 days away    → score = 90,  urgency = "medium", action = "Get a birthday card or plan a surprise"
+10 if isFavorite
Type 3: anniversary
Same logic as birthday but uses anniversary field instead of birthday. Same scoring. Same window.

Type 4: referral_stale
Triggers on referral triangles stuck in a status for too long. The stale thresholds are:

submitted    → stale after 3 days
contacted    → stale after 7 days
connected    → stale after 14 days
in_progress  → stale after 21 days
completed    → no nudge
declined     → no nudge
Scoring:

daysSince = days since statusUpdatedAt (or createdAt)
if daysSince < threshold → no nudge
overdueDays = daysSince - threshold
score = 70 + min(overdueDays × 5, 50)
urgency = overdueDays > 7 → "high" | overdueDays > 0 → "medium" | else "low"
Suggested actions by status:

submitted: "Follow up with [B] to confirm they received the referral"
contacted: "Check if [A] and [B] have connected yet"
connected: "See how things are progressing between [A] and [B]"
default: "Check in on the referral progress"
The nudge name is displayed as "[PersonA name] → [PersonB name]"

Nudges for a referral can be dismissed by setting nudgeDismissedAt on the triangle. Dismissed referrals are excluded from future nudge calculations.

Type 5: engagement
Triggers when someone opens an email, clicks a link, or views a card and that event was tracked within the last 24 hours. Score is always 130 (high priority) because they are actively engaged right now.

score = 130
urgency = "high"
action = "Follow up now while they're engaged"
reason = "[Name] just opened your email from your referral introduction"
Engagement nudges are dismissed by setting dismissedAt on the engagement_events row.

PART 4 — REESE AI ASSISTANT
Persona
Name: Reese
Role: "Your Navigator"
Tone: Calm, grounded, human. Not corporate. Not robotic. Not salesy. Short sentences. Clear thinking. Practical suggestions.

Reese is not a chatbot. Reese is a co-pilot who knows the user's contacts, clients, and referrals and helps the user stay on top of relationships without pressure.

Core Spine (All Features Inherit This)
CATCH — Fast moment capture before it disappears
KEEP  — Intentional decision to promote a catch into a permanent contact
HOLD  — Maintaining full human context over time
For Living Clients: CATCH · KEEP · MANAGE · HOLD
For ReferMe: CATCH · KEEP · REFER · HOLD

Personas / Roles
Reese has four internal operating modes that shape its prompt:

extractor — Pulling structured data from a raw capture (card scan, voice note, etc.). Be accurate and minimal. Do not guess.
writer — Drafting or polishing a communication. Warm and natural. Preserve the user's intent.
analyzer — Summarizing and organizing. Faithful to what's there. Suggest hats carefully.
assistant — General navigation. Default helper mode. Catch/Keep/Hold thinking.
What Reese Does in ReferMe
Identifies referral opportunities from context: "I noticed Sarah might need an accountant — you have Mark who does that. Want me to connect them?"
Walks through referral creation conversationally: "Who needs something? Who can help?"
Always confirms before acting (ask-before-acting workflow): proposes the referral, user approves, then Reese creates it
Checks in on open referrals: "How did the intro between Sarah and Mark go?"
Spots signals in notes: "looking for", "needs a", "should meet", "could use"
What Reese Does in Living Clients
Helps identify which hat to assign a contact to
Suggests pipeline stage updates based on recent notes
Drafts follow-up messages that match the hat's tone
Warns when C is crossing context between hats
If context is unclear: asks "Which hat are we in?"
What Reese Does with Nudges
Surfaces today's nudge list conversationally
Suggests what to say when reaching out
Helps draft messages for birthday nudges, referral check-ins, and overdue contacts
Tracks engagement events: "Sarah just opened your email from the referral you sent — good time to follow up"
AI API Endpoints
POST /api/reese/chat                          — Send a message, get a reply
GET  /api/reese/conversations                 — List past conversations
GET  /api/reese/conversations/:id/messages    — Messages in a conversation

The ask-before-acting pattern: when Reese wants to create, update, or delete anything, it proposes the action in plain English and waits for the user to say "yes" or "go ahead" before executing.

PART 6 — QUIET HOURS
Users can configure quiet hours. When quiet hours are active, notifications (email, SMS) are queued and sent when the window opens rather than immediately.

quietHoursEnabled   boolean
quietHoursStart     text — "21:00" (HH:MM format)
quietHoursEnd       text — "08:00"
quietHoursTimezone  text — IANA timezone string, e.g. "America/New_York"
Apply this to all outbound referral notifications and status updates.

PART 7 — DESIGN PRINCIPLES
Mobile-first, compact. Default to tight, clean views that expand when more detail is wanted. Most users are on phones.
No red for errors (unless critical stop). Use amber/orange for warnings and destructive actions.
No "Uncategorized" labels. Use "Needs Review" instead.
Collapsible sections everywhere. Users should be able to collapse/expand all major sections.
No pressure language. Reese and all UI copy should never imply urgency, force progression, or use sales language.
People over data. Every record is a human being, not a lead or an entry.

How GPS tracking works (build this exactly):

// Constants
MIN_SPEED_MPS = 0.9      // ignore points below 0.9 m/s (stationary drift)
MIN_ACCURACY_M = 100     // ignore points with accuracy worse than 100m
// Storage: localStorage key "active_trip"
// Persists across page reloads so a trip survives navigation
{
  startTime: number,       // timestamp (Date.now())
  totalMiles: number,      // running total
  startLocation: string | null,
  waypoints: [{ lat, lng, ts }]
}
On start:

Call navigator.geolocation.getCurrentPosition() — triggers permission prompt
If denied → show error "Location access denied — check browser settings to allow location"
If granted → initialize store with first waypoint, save to localStorage, set state = "tracking"
Reverse-geocode the start location using https://nominatim.openstreetmap.org/reverse?lat=X&lon=Y&format=json — extract address.road + city/town — save as startLocation (async, non-blocking)
Start navigator.geolocation.watchPosition() loop
On each GPS update:

Skip if accuracy > 100m
Skip if speed < 0.9 m/s (if speed is available)
Calculate haversine distance from last waypoint
Skip if distance < 5m (0.005 km) — filters micro-jitter
Add distance to totalMiles, push new waypoint, save to localStorage
Haversine formula:

function haversineKm(lat1, lng1, lat2, lng2) {
  const R = 6371;
  const dLat = (lat2 - lat1) * Math.PI / 180;
  const dLng = (lng2 - lng1) * Math.PI / 180;
  const a = Math.sin(dLat/2)**2 + Math.cos(lat1*Math.PI/180) * Math.cos(lat2*Math.PI/180) * Math.sin(dLng/2)**2;
  return R * 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1-a));
}

Stop the watchPosition watcher and the elapsed-time timer
Clear localStorage
Return { miles, startLocation, durationMinutes, tripDate } for saving to the mileage log
Minimum miles = 0.1 (prevent 0-mile saves)

Listen to document.visibilitychange
When page goes hidden → stop watchPosition (saves battery)
When page comes back visible → reload store from localStorage, restart watchPosition and timer
