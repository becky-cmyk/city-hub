You are continuing work INSIDE the existing City Hub / Charlotte Hub application.

DO NOT build a separate platform.
DO NOT create a new SaaS.
DO NOT rebuild the hub.
DO NOT generalize into a full event enterprise suite.

You are extending the existing Events capability so events can support real organizer operations inside the current hub ecosystem.

This prompt assumes Prompt 1, Prompt 2, and Prompt 3 already exist:
- event model
- event micro-sites
- invitations / RSVP
- public / unlisted / private visibility
- event discovery layer
- recurring event series
- venue event engine
- Presence integration
- Pulse integration

Now build the **Event Back Office** with sponsor management, vendor management, and internal operational workflows.

==================================================
PRIMARY GOAL
==================================================

Allow organizers and venues to manage the business side of events inside the hub.

This includes:
1. sponsors
2. vendors / exhibitors
3. event partner records
4. public sponsor/vendor display on event pages
5. internal workflow tracking
6. conversion of sponsors/vendors into hub opportunities and Presences later

This is NOT just for public display.
This is also an internal organizer tool.

==================================================
PHASE 1 — SPONSOR MODEL
==================================================

Create an EventSponsor model.

EventSponsor
- id
- event_id (nullable if attached at series level)
- event_series_id (nullable)
- presence_id (nullable if sponsor already has Presence)
- sponsor_name
- sponsor_contact_name (nullable)
- sponsor_email (nullable)
- sponsor_phone (nullable)
- sponsor_website (nullable)
- sponsor_logo_url (nullable)
- sponsor_tier
    title
    presenting
    gold
    silver
    bronze
    supporting
    community_partner
    custom
- sponsor_tier_label (nullable)
- status
    prospect
    invited
    confirmed
    declined
    past
- notes (nullable)
- display_publicly (boolean default true)
- created_at
- updated_at

Rules:
- sponsor can exist without an existing Presence
- if sponsor later becomes a Presence, keep linkable relationship
- sponsors can attach to a single event OR an event series

==================================================

Create an EventVendor model.

EventVendor

- vendor_contact_name (nullable)

- vendor_category (nullable)

- status
    applied
    invited
    approved
    declined
    confirmed
    past
- notes (nullable)

- created_at
- updated_at

- vendors can exist before they become Presences in the hub
- if they are already a Presence, link them

==================================================

Extend the event dashboard.

Route example:
- /dashboard/events
- /dashboard/events/{id}
- or the route structure most consistent with the existing app

For each event or series, add tabs/sections for:

Overview
Invitations / RSVP
Sponsors
Vendors
Files / Assets (simple placeholder if needed)
Analytics (lightweight)
Archive / History

Sponsor section should allow:
- add sponsor manually
- edit sponsor
- update tier
- mark status
- toggle public display

Vendor section should allow:
- add vendor manually
- approve/reject vendor
- edit booth/category
- toggle public display
- link to existing Presence if available

Keep UI functional, simple, and consistent with the current admin/dashboard patterns.

An organizer or venue can:
