BUILD A NEW SUPER ADMIN ENGINE IN THE EXISTING CITY HUB PLATFORM CALLED:

Public-facing name:
Enter to Win

Admin / internal name:
Enter to Win Engine

PLACE IT UNDER:
Super Admin → Engagement Hub → Enter to Win

IMPORTANT:
This is NOT a lightweight MVP. Build the full backend/admin engine discussed below so it can support Hub-owned giveaways, business/client giveaways, recurring fishbowl-style QR campaigns, weighted bonus entries, automated Charlotte-driven winner workflows, previews, publishing, updates, and reusable QR infrastructure.

The build must follow the City Hub architectural rule:
single source of truth → multiple discovery surfaces
Do NOT create disconnected giveaway tools, duplicate objects, or one-off form systems.

==================================================
1. HIGH-LEVEL PURPOSE
==================================================

Build a full Super Admin giveaway / lead capture / engagement / winner follow-up engine that allows:

- CLT City Hub to run its own giveaways
- businesses/clients to use the same engine as an add-on later
- recurring “fishbowl” QR entries at stores/events without replacing QR signage
- one entry per email per giveaway
- Reader membership matching/creation
- Charlotte as the voice of the full automation process
- controlled business lead-sharing only when the entrant explicitly opts in
- weighted/stackable bonus entries defined per giveaway by admin
- visual wheel-based winner selection
- public results pages with masked names by default
- automatic content/story upgrades after winner consent/approval
- full backend previews before publish
- full update controls after publish
- issue handling routed to the admin inbox

==================================================
2. ADMIN NAVIGATION / BACKEND STRUCTURE
==================================================

Create the following backend structure:

==================================================
3. CORE GIVEAWAY LIFECYCLE / STATES
==================================================

Support these giveaway statuses:

- draft
- scheduled
- live
- closed
- draw_pending
- completed
- archived

Behavior:
- all setup begins in draft
- nothing public is live until published
- no draw before giveaway ends
- entries stop at close/end
- timezone stays local to the business or metro
- no advanced timezone conversion logic needed

==================================================
4. GIVEAWAY TYPES
==================================================

Support both:
- one-time giveaway
- recurring / reusable giveaway

Recurring / reusable giveaway behavior:
- same QR can stay in place at a business/location
- admin can reuse the giveaway setup/campaign shell
- admin can reattach or roll the recurring campaign without replacing physical QR signage
- QR continuity must be preserved where recurring toggle is enabled

5. SUPER ADMIN GIVEAWAY BUILDER
==================================================

Create a full Giveaway Builder with these sections/tabs:

A. CORE
- public title
- internal name
- slug
- owner type (hub_owned, business_owned)
- assigned business/client nullable
- giveaway type (one_time, recurring)
- start date/time
- end date/time
- status
- internal admin notes

B. PRIZE SETUP
- one or many prizes
- prize title
- description
- image
- quantity
- value estimate optional
- associated business/location if relevant

C. ENTRY RULES
Locked rules:
- one entry per email per giveaway
- full name required
- email required
- phone required
- zip optional
- Reader membership required
- if existing Reader found by email, match account and prompt login
- if no Reader exists, create/match free Reader flow
- email verification required for valid entry
- honeypot enabled
- terms text required
- giveaway detail text visible on giveaway page and form
- explicit opt-in toggle for sharing with business/store (NOT automatic)

D. BONUS ENTRY RULES
Each giveaway must have a configurable bonus section:
- enable bonus entries yes/no
- admin can add any number of bonus actions
- each action has:
  - action name
  - description
  - weight / extra entry value
  - stackable yes/no
  - active yes/no
  - internal verification/notes field
- stacking allowed across actions
- admin controls weights per action/tier
- if a user has already entered, they should be shown ways to earn bonus entries
- duplicate entries are void if there is no valid bonus path attached

E. QR / SOURCE RULES
Important locked rule:
- the giveaway QR is the only QR for entry to that giveaway
- no multi-QR duplicate entry paths for the same giveaway entry
- QR is generated when giveaway is published
- QR must always remain accessible from the giveaway record
- QR must be downloadable
- QR link must be copyable
- QR preview must be visible in admin
- QR branding required:
  - include CLT logo
  - include CLTCityHub.com under the code
- support recurring/reusable QR behavior when recurring toggle is on

F. DRAW RULES
- visual wheel enabled
- supports:
  - single winner
  - multiple winners
  - alternates
  - redraws
- 48-hour claim window
- no auto-promote final public change without admin review
- no drawing before giveaway end
- weighted entries must be respected
- draw pool only includes valid eligible entries

G. WINNER CONTENT / PERMISSION RULES
- default public display: first name + last initial only
- after winner gives permission and admin approves, auto-upgrade public display to:
  - full name
  - quote
  - image / likeness if provided
- support collection of:
  - permission to use name
  - permission to use quote
  - permission to use photo/likeness
  - permission to tag/mention business/location
- support optional review/comment about the location/business/event they won from

H. AUTOMATIONS
Charlotte is the voice of the entire process.

Automation toggles/templates for:
- verification email
- winner congratulations email
- reminder 1
- reminder 2 / final notice
- alternate notification
- winner content request / claim form intro
- admin draw summary
- issue alerts routed to inbox

7. ENTRY FLOW / READER MATCH LOGIC
==================================================

Locked entry behavior:

Public form fields:
- full name
- email
- phone
- zip optional

Flow:
1. user lands on giveaway entry page from the giveaway QR / page
2. minimal form shown
3. honeypot checked server-side
4. email is checked against existing Reader/member/contact
5. if existing Reader account is found:
   - prompt login / match account
   - attach entry to existing Reader/contact
6. if no Reader exists:
   - create/match free Reader flow
   - send verification email
7. only after email verification is completed does the entry become valid/verified
8. one entry per email per giveaway
9. if duplicate attempt and no valid bonus path is configured, mark duplicate/void and do not count
10. if already entered and bonus actions are enabled, show “ways to earn bonus entries”

Locked identity rule:
- email is the primary unique key for entry enforcement
- phone is secondary/supporting validation only
