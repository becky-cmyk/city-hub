PHASE 1 SCOPE

==================================================
A. FIELD CAPTURE SYSTEM
==================================================

There are multiple field capture input methods already conceptually in use and they should all route into ONE shared backend pipeline.

Capture input methods to support:

- Business Card / Photo
- From My Phone
- QR Code / vCard
- Manual Entry
- Voice Note
- Handwrite

IMPORTANT:
Field Capture must support 3 distinct outreach origins. This is critical and must be structured in the data model and processing flow, not left to freeform interpretation later.

Required capture_origin values:

1. met_in_person
   Meaning:
   - I actually met this person / business owner / rep
   - I exchanged contact details or spoke with them directly

2. stopped_by_location
   Meaning:
   - I visited or stopped by the location
   - I may not have met them directly
   - I want the outreach to reflect that I noticed the business in person

3. found_business_card
   Meaning:
   - I picked up a card, flyer, brochure, rack card, or printed info
   - I did not necessarily meet them directly
   - outreach should reflect passive discovery

REQUIRED FIELD CAPTURE PIPELINE

Every capture method should route through the same backend logic:

Capture
→ choose or infer capture_origin
→ extract data
→ normalize data
→ match against existing records
→ create or update contact
→ create or update business / organization / venue entity
→ assign hub
→ assign category/type
→ create or update listing
→ create Opportunity Radar record
→ prepare activation-ready outreach draft based on capture_origin

Data extraction targets should include as much of the following as possible:

- person name
- business/organization name
- phone
- email
- website
- address
- social links
- category / business type
- venue potential
- organization type
- notes / context from capture

Must include duplicate/matching logic to reduce duplicate contacts/listings if already present.

Field Capture should be treated as a real-world intelligence intake layer.
This is not just a contact form. It is a way for the operator to seed people, businesses, organizations, and venues while networking and moving through the city.

Required output of field capture:
- contact record
- entity record (business / organization / venue / person as appropriate)
- listing created or updated where applicable
- hub assignment
- opportunity radar entry
- activation-ready outreach draft

Do NOT build heavy follow-up sequence logic yet.
Just make sure the first capture-to-activation setup is structurally correct.

==================================================
B. OUTREACH DRAFTS BASED ON CAPTURE ORIGIN
==================================================

The system must support 3 different default outreach message frames tied to capture_origin.

1. met_in_person
Default framing:
- Great meeting you
- I added your business/info to the hub
- Please confirm I set it up correctly

2. stopped_by_location
Default framing:
- I stopped by your location today
- I noticed / was impressed by your business
- I added your business to the hub
- Please confirm the details

3. found_business_card
Default framing:
- I came across your business card today
- I thought your business sounded like a great fit for the hub
- I went ahead and listed it on the hub
- Please confirm the information

IMPORTANT:
These should be generated as structured email draft types controlled by capture_origin.
Do NOT make the operator manually rewrite these each time.
Do NOT leave this as vague prompt logic.
Build this as a deterministic mapping.

Please audit where this could live in the existing messaging / draft / outreach systems if those already exist.

==================================================
C. CHARLOTTE DAILY REPORT / CHARLOTTE INTELLIGENCE
==================================================

Build or tighten a daily operator briefing layer called Charlotte Report (or equivalent if something similar already exists).

Charlotte should report TO the operator instead of forcing the operator to dig through many admin screens.

This daily briefing should include only the most important operational blocks:

1. Hub Signals
- hottest neighborhoods / hubs
- rising hubs
- dormant hubs
- seeded-but-inactive hubs

2. New Opportunity Activity
- new businesses found
- new organizations found
- new venue prospects
- new field captures
- press/chamber/announcement opportunities
- seeded businesses ready for activation

3. Top Contactable Leads
Surface the strongest leads only, such as:
- venue prospects
- activation-ready businesses
- high-value organizations
- visibility / ad / banner prospects where applicable
- story-first candidates

4. Exceptions / Needs Human Review
Only show:
- duplicate conflicts
- failed extraction
- ambiguous matches
- claim/confirmation problems
- risky content / risky classification
- important system issues Charlotte could not resolve confidently

5. Suggested Actions for Today
Examples:
- which hub to visit
- which venue prospects to contact
- which captured businesses need activation outreach
- which listings are ready for confirmation push

IMPORTANT:
Charlotte Report should be concise, action-oriented, and useful for a single operator.

==================================================

Build or tighten one unified actionable board called Opportunity Radar.

This is NOT a generic CRM list.
It is the operator’s action board for discovered businesses, venues, organizations, and opportunities from all sources.

Opportunity Radar should collect records from:

- field capture
- Google Places / seeded business import
- press releases
- chamber announcements
- AI discovery
- story interviews / profile intelligence
- other relevant system discovery sources already in place

Each Opportunity Radar row should show at minimum:

- name
- entity type
- hub / neighborhood
- source
- contact info available
- activation status
- why it was surfaced
- recommended next step
- priority

Suggested entity/opportunity types may include:
- business
- venue
- organization
- claim candidate
- activation-ready
- story-first candidate
- visibility/ad prospect
- partnership candidate

IMPORTANT:
All discovered entities should not disappear into background data tables. They should land in Opportunity Radar in a way that is actually usable for the operator.

If something similar already exists, audit it and tighten it rather than duplicating it.
