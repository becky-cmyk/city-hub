Work inside the existing City Hub codebase.

Goal:
Build the foundation pass for the unified RSS/imported content routing system.

This is NOT a Pulse rebuild.
This is NOT a UI redesign.
This is NOT a flattened tag cleanup.

This task exists to make sure RSS/imported content supports true multi-layer category + geo routing so the ecosystem can distribute content correctly later.

CRITICAL GUARDRAIL — DO NOT FLATTEN THIS:

The system we are building is explicitly MULTI-CATEGORY and MULTI-GEO.

Do NOT reduce this to:
- one category per item
- one generic tag bucket
- one flat location field
- one simplified “news” fallback

That would recreate the current problem.

We are intentionally building a richer routing structure because the platform depends on:
- core categories
- subcategories
- micro-category readiness
- primary geography
- secondary geography
- hub / county / metro relationships

If this is flattened, the platform loses the exact behavior it is supposed to support.

--------------------------------------------------
PART 4 — MULTI-LAYER GEO FIELDS ON RSS/IMPORTED CONTENT
--------------------------------------------------

Enhance rss_items (or the appropriate imported-content source table) so each item can support multiple geography dimensions.

At minimum support:
- geo_primary
- geo_secondary
- county
- metro
- hub / micro hub if resolvable
- raw location/venue signal if available

Rules:
- geography is NOT a category
- do not stuff location into category tags
- do not flatten geography into one generic zone string
- do not leave location null when broader geography can be inferred

Geo fallback model:
1. exact/hub/micro-hub if resolvable
2. county
3. metro

Every imported item should end with enough geo structure to support local-first distribution later.

Also preserve the possibility that one item may relate to more than one geography:
- primary geography = where it belongs first
- secondary geography = where else it is relevant

This is required. Do not flatten to single-location only.

--------------------------------------------------
PART 5 — SIMPLE PROCESSING METADATA (NOT OVERBUILT)
--------------------------------------------------

Do NOT create an over-engineered 11-status state machine.

Instead support two cleaner concepts:

A. publish_status
Examples:
- DRAFT
- REVIEW_NEEDED
- APPROVED
- PUBLISHED
- SUPPRESSED
- ARCHIVED

B. processing_stage
Examples:
- INGESTED
- REWRITTEN
- CLASSIFIED
- GEO_ASSIGNED
- ROUTED

This allows us to track pipeline progress without making admin control unmanageable.

If a JSON checkpoint/log is more appropriate than a giant enum, that is acceptable.
