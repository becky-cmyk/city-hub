You are Replit AI working inside our existing CLT Connects / Charlotte City Hub codebase.

GOAL
We previously built this as a directory + publication. We now want the USER EXPERIENCE to feel like TikTok/Instagram/Facebook/Nextdoor in MOBILE, but it is NOT social posting. It is a highly-visual, curated INFORMATION FEED that is easy to scroll. The feed is GEO-FIRST and TAG-FIRST.

KEY CONCEPT
All our “categories, topics, locations, neighborhoods, ZIPs, businesses, organizations” are treated as structured “hashtags” (tags). Users scroll a mixed feed of real local content. Each card shows a primary tag (topic) + location tag and is tappable: tapping a tag re-filters the feed to that tag combination.

CONSTRAINTS
- Do NOT rebuild or discard existing data. Reuse current directory/publication data (businesses, organizations, events, articles/pulses, etc.).
- This is a PRESENTATION + QUERY shift, not a replatform.
- Mobile-first UI. Desktop should feel like a widened mobile feed.
- Ads/sponsored placements must appear as feed cards (clearly labeled “Sponsored”), targeted by tag + geo, not banner ads.
- Keep SEO strong: tag-based URLs should be indexable and internally linked.

WHAT TO BUILD (PHASE 1: FUNCTIONAL MVP)
1) TAG MODEL (database + normalization)
   - Create a normalized tags system IF we don’t already have one:
     - tags table: id, slug, label, type (location|topic|entity|status), parentTagId nullable (for location hierarchy), synonyms optional
     - content_tags table: contentType, contentId, tagId
   - Do NOT migrate everything at once. Provide a safe approach:
     - Backfill tags from existing fields (zip, neighborhood, category, etc.) into tags + content_tags
     - Keep existing columns for now; tags become the new query layer.

2) CONTENT UNIFICATION (“feed items”)
   - Create a server-side “feed item” projection that unifies multiple content types:
     - Event
     - Business/Presence
     - Organization
     - Pulse/Article (if exists)
     - SponsoredPlacement (ad card)
   - Each feed item should include:
     id, type, title, subtitle, imageUrl, primaryTag (topic), locationTags (zip/neighborhood/city), entityTag (business/org if applicable),
     startDate/endDate (for events), createdAt, sponsored boolean, sponsorshipMeta if applicable,
     url (deep link), and a small “why shown” explanation (for admin/debug).

3) FEED API
   - Implement endpoints:
     - GET /api/feed?geoTag=<slug>&topicTag=<slug>&context=<trending|nearby|new>&page=&limit=

     - GET /api/tag/<slug> (tag metadata + counts)
   - Feed filtering rules:
     - Default geoTag: auto-detected zip if we have it, otherwise city.
     - topicTag optional; when selected, filter to items that have BOTH geoTag (or descendant) AND topicTag.
     - Always mix types in the feed (not separate sections).
