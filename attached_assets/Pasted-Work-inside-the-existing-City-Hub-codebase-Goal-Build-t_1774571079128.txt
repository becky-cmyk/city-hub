PART 2 — FULL ARTICLE GENERATION RULE
--------------------------------------------------

For RSS/imported story content:
Charlotte AI must ALWAYS generate a full local article, not just an excerpt.

Requirements:
- generate a real article body in local/story format
- target approximately 400–700 words unless existing app logic requires slight variation
- article should feel original and locally useful
- article should include source attribution
- article should include original source URL
- article should support image/source credit metadata
- do NOT simply store a teaser plus outbound link as the main output
- do NOT copy/paste source text verbatim into the public article body

Store source metadata cleanly, including:
- source publication / source name
- original source URL
- source image URL if available
- image credit / image source if available
- ingestion timestamp

If the current pipeline only stores summary/excerpt behavior, refactor it so the system stores and exposes the fuller article output.

--------------------------------------------------
PART 3 — MULTI-LAYER CLASSIFICATION
--------------------------------------------------

Every imported content object must support MULTIPLE routing layers, not single-tag classification.

Required classification dimensions:
1. content_type
   - story
   - event
   - job
   - marketplace
   - business update
   - community update
   - other supported type

2. category_core
   - top-level category

3. category_sub
   - one level deeper than core

4. geo_primary
   - where this belongs first

5. geo_secondary
   - where else this is relevant
   - always include metro relationship when appropriate

6. related_entity
   - business / organization / venue / person if confidently detected

7. source_type
   - rss / imported / ai-assisted / etc.

Do not flatten category and geography into one tag system.
Keep topic/category separate from geography.

--------------------------------------------------
PART 4 — GEO-FIRST ROUTING
--------------------------------------------------

Geo precision matters more than broad generic placement.

Implement routing support for:
- exact location if available
- venue-based location
- hub / micro hub
- county
- metro fallback

Rules:
- every eligible imported item must end with a geo_primary
- if exact hub cannot be resolved, use county
- if county cannot be resolved, use metro fallback
- geo_secondary should support broader relevance where appropriate
- do not leave imported items as location-null if routing can infer a broader geography

This system must support the later local-first Pulse behavior where current location and radius determine visibility.

Do not build radius ranking yet in this prompt.
Just make the content object geo-ready.

PART 6 — HYBRID REVIEW QUEUE
--------------------------------------------------

Build a hybrid queue model.

Not all RSS/imported content should auto-publish.
Not all content should require manual review.

Support queue decisions like:
- AUTO_PUBLISH
- REVIEW_REQUIRED
- SUPPRESS

Send to REVIEW_REQUIRED if any of these are missing or weak:
- no usable category
- no usable geo
- low rewrite confidence
- missing source attribution
- missing image/source credit where required
- policy ambiguity
- low local relevance
- unclear entity/venue match

Allow auto-publish only when the item passes minimum routing confidence.

Important:
RSS/imported items must no longer silently skip the inbox/queue if they need review.
We need visibility into where they are landing.
