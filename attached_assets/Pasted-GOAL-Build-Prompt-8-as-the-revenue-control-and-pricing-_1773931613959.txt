GOAL:
Build Prompt 8 as the revenue control and pricing enforcement layer across the platform and all metros.

This prompt ensures:
- pricing is controlled centrally (platform-first)
- metros inherit pricing correctly
- Cora can suggest changes but NOT apply them
- Stripe products/prices stay aligned
- entitlements and access are consistent
- no revenue chaos as you scale metros

IMPORTANT:
- DO NOT break existing Charlotte pricing
- DO NOT duplicate Stripe products unnecessarily
- DO NOT allow per-metro pricing drift (for now)
- DO NOT auto-change pricing
- DO NOT auto-create Stripe products without approval

==================================================
PART 1 — DEFINE PLATFORM PRICING AS SOURCE OF TRUTH
==================================================

Create (or confirm) a platform-level pricing table:

platform_products
- id
- name
- description
- product_type (subscription, one_time, add_on)
- tier (free, verified, charter, enhanced, etc.)
- scope (platform, metro, hub, creator, business)
- stripe_product_id
- is_active
- created_at
- updated_at

platform_prices
- id
- product_id
- billing_interval (monthly, annual, one_time)
- price_amount
- currency
- stripe_price_id
- is_active
- created_at
- updated_at

RULE:
Platform pricing is the ONLY place where base pricing is defined.

==================================================
PART 2 — METRO PRICING INHERITANCE
==================================================

Create table:

metro_pricing_overrides
- id
- metro_id
- product_id
- override_type (none, discount, override_price, custom)
- override_value (nullable)
- is_active
- created_at
- updated_at

DEFAULT RULE:
- metros inherit platform pricing
- overrides are OFF by default

For now:
- Becky is using SAME pricing across metros
→ enforce inheritance by default

==================================================
PART 3 — ENTITLEMENTS LAYER (ACCESS CONTROL)
==================================================

Create/extend entitlements table:

user_entitlements
- id
- user_id
- product_id
- metro_id (nullable)
- scope (platform, metro, hub)
- status (active, inactive, canceled, past_due)
- start_date
- end_date (nullable)
- source (stripe, manual, promo)
- created_at
- updated_at

Purpose:
- control access to:
  - business features
  - creator features
  - hub placement
  - content tools
  - future add-ons

DO NOT rely on Stripe alone for access logic.

Cora should be able to:
