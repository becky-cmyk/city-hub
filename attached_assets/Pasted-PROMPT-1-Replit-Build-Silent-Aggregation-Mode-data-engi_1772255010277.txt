PROMPT 1 (Replit) — Build “Silent Aggregation Mode” data engine: Business Filings + Multifamily/Apartments + Language/Demand overlay + Lead lists (internal only)

CONTEXT / GOAL
We are building the Metro Intelligence Engine in “silent aggregation mode” (no public dashboards, no selling yet). This system must:
1) Collect + normalize data for:
   A) Business Filings (NC focus first; structure supports any state later)
   B) Multifamily / Apartment communities
   C) Language/Demand overlay (English vs Spanish usage + search intent)
2) Turn these into internal “lead lists” and a unified “Signals” feed for later monetization.
3) Be multi-tenant by metro_id and zip_id (licensable).
4) Collect quietly now; expose only a private admin page and internal APIs.

NON-NEGOTIABLES
- Do NOT build public dashboards yet.
- Do NOT change existing public UX.
- Build as a modular data spine that can be duplicated for other metros/states.
- Capture data even if site traffic is low; log consistently.
- All new tables must include: id, metro_id, zip_id (if applicable), source, source_url (nullable), created_at, updated_at.

B) INGESTION (Manual-first, automation-ready)
Create server routes + services:

1) POST /api/admin/ingest/business-filings
- Accepts an array of filings (manual import for now)
- Normalizes: trims names, uppercases state_code, parses dates, resolves zip -> zip_id when possible
- Upserts by filing_external_id when provided; otherwise insert with best-effort dedupe (business_name + filing_date + registered_address)
- Also writes a signals_feed record per new filing

2) POST /api/admin/ingest/multifamily
- Accepts an array of multifamily properties
- Normalizes address, parses unit_count/rent ranges, resolves zip -> zip_id
- Upserts by (property_name + address) as a composite dedupe
- Also writes a signals_feed record per new property

3) Background job hooks (implement job scaffolding, but do not scrape yet)
- Create a jobs folder + placeholders:
  - job_pull_business_filings_nc.ts (disabled)
  - job_pull_multifamily_listings.ts (disabled)
Include environment flags to enable later:
- ENABLE_FILINGS_PULL=false
- ENABLE_MULTIFAMILY_PULL=false

C) DATA CAPTURE FROM SITE (Language/Demand overlay)
Implement lightweight logging middleware:

1) On page view events (server-rendered or API called on client):
- POST /api/log/page-view
Body: { metroId, zipCode(optional), pageType, pageRefId(optional), language }

2) On language toggle:
- POST /api/log/language-toggle
Body: { metroId, zipCode(optional), pageType, pageRefId(optional), language }

3) On search submit (if search exists):
- POST /api/log/search
Body: { metroId, zipCode(optional), language, queryText, categoryId(optional) }

All three write to language_usage_log. Keep it fast (no blocking UX).
