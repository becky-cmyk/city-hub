==================================================
PRIMARY PRINCIPLE
==================================================

The City Hub is sold as a layered ecosystem.

That means:
- some things are included in the base level
- some things are recurring add-ons for people who want ongoing use
- some things can be done occasionally with credits
- some things are premium visibility/distribution opportunities
- some things are internal/operator-only functions

The system must support all of these without requiring custom handling every time.

==================================================
PART 2 — PACKAGE INTENT
==================================================

Use the following business intent for each level.

FREE
Purpose:
- basic discovery presence or participation entry
- low-friction entry into the hub
- mostly viewing / limited participation
- does not unlock serious commercial use

VERIFIED
Purpose:
- entry paid presence
- verified participation
- can begin using selected posting and visibility functions
- includes limited usage / limited credits

CHARTER
Purpose:
- stronger business/brand presence
- more visibility
- more participation rights
- better placement / better inclusion in hub activity
- includes more credits and better distribution eligibility

ENHANCED
Purpose:
- serious active participants using the hub as an ongoing channel
- more tools, more visibility, more distribution
- suitable for businesses/brands/operators wanting stronger recurring use

ENTERPRISE
Purpose:
- top-tier presence or partner-level usage
- broad access, expanded rights, premium visibility, multi-location or more advanced network use where appropriate

PART 4 — PACKAGE MAPPING RULES
==================================================

Apply this logic as the starting framework.

A. FREE
Should generally allow:
- basic profile/presence if already supported
- limited discoverability
- view hub content
- possibly submit/request participation where moderation exists
- no major commercial posting rights by default

Should generally NOT include:
- unlimited event posting
- unlimited job posting
- storefront/marketplace selling
- premium placement
- broad media distribution
- advanced boosts

B. VERIFIED
Should generally allow:
- verified listing/presence
- access to selected publishing/posting actions
- limited event/job/marketplace actions depending on module
- small monthly credit allocation
- low-to-moderate visibility participation

Should not automatically include all premium placements or expanded media distribution.

C. CHARTER
Should generally allow:
- stronger presence
- more included actions
- higher posting limits
- more credits
- better placement eligibility
- eligibility for stronger pulse/content exposure
- stronger promotional participation

D. ENHANCED
Should generally allow:
- larger included action set
- broader recurring usage rights
- stronger visibility/distribution
- more advanced promotional/feature tools
- larger credit pool
- broader module access

E. ENTERPRISE
Should generally allow:
- highest visibility and inclusion rights
- expanded distribution eligibility
- advanced placement options
- multi-location / multi-entity support where relevant
- top analytics/reporting access if supported
- premium handling paths if already present in the app

PART 5 — ADD-ON STRATEGY
==================================================

Add-ons are for users who want a function as an ongoing lane, not just a one-time use.

Build package mapping to support add-ons such as:
- job board access / recurring job posting
- marketplace storefront
- event host / recurring event posting
- expert profile
- creator network / creator tools
- featured distribution
- premium placement
- video channel
- podcast channel
- multi-hub presence

Important business rule:
A user may not need the recurring add-on if they only want occasional use.
That is where credits come in.

So:
- add-on = ongoing functional lane
- credit = occasional action or one-time use

==================================================
PART 6 — CREDIT LOGIC FOR PACKAGES
==================================================

Map credits as flexible one-time usage.

Examples of actions that may be credit-eligible:
- post one event
- post one job
- list one classified
- add one marketplace listing
- push content into a featured lane
- boost a story/post
- request social distribution
- unlock one premium placement action
- request video/podcast distribution if allowed

Important rule:
Credits should allow occasional access without forcing the user into a recurring add-on.

Example:
- business does not subscribe to Job Board add-on
- business can still use credits to post a one-off job

Example:
- seller does not subscribe to Marketplace Storefront
- seller can still spend credits to post a one-time item

Use this framework throughout the package mapping.
